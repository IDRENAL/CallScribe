# CallScribe

Запись звонков (микрофон + системный звук) и их транскрипция в текст через
[faster-whisper](https://github.com/SYSTRAN/faster-whisper). Кросс-платформенно:
Windows 10/11 и Linux (Ubuntu 22.04+/Debian 12+).

## Возможности

- Одновременный захват микрофона и системного звука (loopback) в стерео-WAV (L=mic, R=система)
- Транскрипция моделью `large-v3` (CPU с мультипроцессингом или GPU при наличии CUDA)
- Два режима точности (см. ниже)
- Стенограмма в Markdown + JSON с разделением спикеров
- Диаризация (кто говорит) для одиночных дорожек mp4 — опционально, pyannote
- LLM-выжимка разговора (тезисы, решения, задачи) — локально через Ollama
- Экспорт стенограммы и выжимки в один `.docx`
- Веб-интерфейс (FastAPI + WebSocket) с живым логом
- Только локальные файлы, без баз данных и облака

## Режимы точности (`mode` в `config.json`)

- **`accurate`** (по умолчанию) — микрофон и системный звук расшифровываются
  **раздельно**. Не теряются реплики при наложении речи, сохраняется контекст,
  мягкий VAD не режет тихую речь, включены анти-галлюцинационные пороги. В
  стенограмме реплики помечены спикерами (`Я` / `Собеседник`). Длинные каналы
  режутся на чанки **только по паузам** (слова не разрезаются) — это загружает
  все ядра, не жертвуя точностью. Медленнее `fast`, но максимально полно.
- **`fast`** — запись сводится в моно и режется на чанки в несколько процессов.
  Быстрее за счёт загрузки всех ядер, но возможны потери слов на стыках чанков
  и при перебивках.

Дополнительно: `compute_type` (`null`=авто / `float32`=точнее на CPU / `int8`),
`vad` (мягкий VAD вкл/выкл), `speaker_labels` (подписи дорожек),
`cpu_workers` (`null`=авто — число CPU-воркеров подбирается по ядрам **и**
свободной ОЗУ; каждый воркер грузит свою копию large-v3 ~3.5 GB, поэтому без
учёта памяти много воркеров могут исчерпать ОЗУ и подвесить систему. Число —
жёсткий потолок, например `2`).

### Выбор устройства (CPU / GPU)

`device` в `config.json`: `auto` (GPU если есть, иначе CPU) / `cpu` / `cuda`.
В веб-интерфейсе есть селектор «Считать на: Авто / CPU / GPU» (GPU неактивен,
если карта не найдена) — выбор применяется к запускаемой задаче. В CLI —
флаг `--device`: `python main.py transcribe call.mp4 --device cpu`.

## Установка

Требуется **Python 3.12+**.

```bash
uv sync
```

(Зависимости описаны в `pyproject.toml`, точные версии — в `uv.lock`.
Если [uv](https://docs.astral.sh/uv/) ещё не установлен:
`curl -LsSf https://astral.sh/uv/install.sh | sh`.)

### Системная зависимость: PortAudio (нужна только для записи)

`sounddevice` требует библиотеку PortAudio. Транскрипция и веб-интерфейс
работают и без неё — она нужна лишь чтобы **записывать** звук.

```bash
# Ubuntu/Debian
sudo apt install libportaudio2

# Windows — PortAudio идёт в составе колеса sounddevice, доп. установка не нужна
```

### GPU-ускорение (NVIDIA, опционально)

При наличии NVIDIA GPU транскрипция идёт на видеокарте (CTranslate2, torch не нужен):

```bash
uv sync --extra gpu          # ставит CUDA-библиотеки (cuBLAS/cuDNN, CUDA 12)
# или: uv pip install nvidia-cublas-cu12 nvidia-cudnn-cu12
```

Библиотеки подгружаются автоматически — вручную `LD_LIBRARY_PATH` настраивать не
нужно. `compute_type` подбирается под карту: на современных GPU — `float16`, на
Pascal (GTX 10xx, где float16 неэффективен) — `int8_float32`. CPU-режим работает
и без этих пакетов.

### Системный звук (loopback)

- **Windows:** установить [VB-Cable](https://vb-audio.com/Cable/) и выбрать «CABLE Output»,
  либо включить «Стерео микшер» в записывающих устройствах.
- **Linux (PulseAudio/PipeWire):** системный звук — это `*.monitor` текущего sink'а.
  Поставь `pulseaudio-utils` (`sudo apt install pulseaudio-utils`) — тогда `setup`
  сам найдёт monitor-источники через `pactl` и предложит нужный (монитор default
  sink — первым). Запись с monitor идёт через устройство `pulse` + `PULSE_SOURCE`,
  поэтому индекс PortAudio для этого не нужен.

## Настройка

```bash
python main.py setup
```

Мастер покажет аудио-устройства и сохранит их ID в `config.json`
(этот файл не коммитится — на каждой машине свои устройства).

## Использование

**Запуск всего одной командой** (Ollama для выжимки + веб-интерфейс, сам откроет браузер):

```bash
make start
# или напрямую:  ./start.sh
# по сети (с ноутбука):  make start HOST=0.0.0.0
```

`start.sh` поднимает Ollama (если установлен — через `systemd --user` или фоном),
дожидается его и запускает UI. Если Ollama нет — UI всё равно стартует, просто
выжимка будет недоступна (транскрипция работает).

Остальные команды:

```bash
python main.py ui                 # только веб-интерфейс на http://127.0.0.1:5000
python main.py run                # запись + транскрипция (CLI)
python main.py record             # только запись (Enter — стоп)
python main.py transcribe a.wav   # транскрибировать готовый WAV
python main.py transcribe call.mp4 # видео/аудио файл (mp4/mkv/mp3/…) — см. ниже
python main.py last               # обработать последнюю запись
```

### Внешние файлы (видео звонка mp4 и др.)

`transcribe` принимает не только WAV: видео и аудио любых форматов
(`mp4`, `mkv`, `mov`, `m4a`, `mp3`, `ogg`, …) декодируются через PyAV — отдельный
ffmpeg ставить не нужно. Аудиодорожка извлекается автоматически.

Нюанс: запись звонка в mp4 — это **одна сведённая дорожка**, поэтому деление по
каналам (`Я`/`Собеседник`) недоступно. По умолчанию выйдет единый текст с
таймкодами. Если включить **диаризацию** (см. ниже), спикеры на такой дорожке
будут размечены (`Спикер 1`/`Спикер 2`/…). Для собственных стерео-записей
CallScribe диаризация не нужна — каналы и так разделены (L=mic, R=система).

В веб-интерфейсе внешние файлы можно просто **перетащить** в зону «Внешний файл»
(drag-and-drop). После загрузки файл выбирается в списке, но обработка **не**
стартует сама — запусти её кнопкой «Транскрибировать» или «Обработать (всё)».
Идущую обработку можно прервать в любой момент кнопкой «■ Остановить обработку»
(результат при этом теряется).

> **CPU:** ориентируйся на ~x3 от длительности звонка (час созвона ≈ 3 часа обработки).
> **GPU:** на уровне GTX 1080 — ~20 минут на часовой звонок.

Модель `large-v3` (~3 GB) скачивается автоматически при первом запуске в `models/`.

## Запись на ноутбуке, обработка на GPU-десктопе

Если ноутбук слабый (или только со встроенной графикой), а тяжёлую транскрипцию
хочется считать на машине с NVIDIA GPU — раздели роли:

1. **На десктопе с GPU** запусти веб-интерфейс, доступный по сети:
   ```bash
   make ui-lan            # слушает 0.0.0.0, печатает адрес вида http://192.168.0.4:5000
   ```
2. **На ноутбуке** запиши звонок (`make record` или своей программой) и открой
   в браузере адрес десктопа. Перетащи запись (mp4/wav/…) в зону «Внешний файл» —
   файл загрузится на десктоп и обработается на **GPU** (`device: auto` → CUDA).
3. Готовая стенограмма сразу показывается в том же окне браузера на ноутбуке.

> **Встроенная графика (Intel/AMD iGPU) для расчёта не подходит:** движок
> CTranslate2 умеет только CPU и **CUDA** (дискретные NVIDIA). Поэтому ноутбук
> здесь — только клиент записи/просмотра, а считает GPU десктопа.
>
> ⚠ У веб-интерфейса нет пароля — поднимай `ui-lan` только в доверенной домашней
> сети (не на публичном Wi-Fi).

## Тесты

```bash
make test          # или: uv run pytest
```

Покрыты чистые функции без обращения к железу и модели Whisper: микширование и
WAV-round-trip (моно/стерео), нарезка по паузам (инвариант «слова не режутся»),
лимит CPU-воркеров по ОЗУ, склейка и дедупликация сегментов, разбор стерео-каналов,
рендер Markdown, загрузка/слияние `config.json`. Тесты быстрые (~1 c), без сети.

## Диаризация (кто говорит) для одиночных дорожек

Для загруженного **mp4/моно** (одна сведённая дорожка) можно разметить спикеров
(`Спикер 1`/`Спикер 2`/…) через [pyannote.audio](https://github.com/pyannote/pyannote-audio).
Собственным стерео-записям это не нужно — там каналы уже разделены.

Диаризация — опциональна (тянет `torch`) и по умолчанию **выключена**:

```bash
uv sync --extra diarize        # установить pyannote.audio + torch
```

Плюс нужен бесплатный **токен Hugging Face** и принятые условия модели
(на странице `pyannote/speaker-diarization-3.1` нажать «Agree»). Затем в
`config.json`:

```jsonc
"diarize": {
  "enabled": true,
  "model": "pyannote/speaker-diarization-3.1",
  "hf_token": "hf_…",            // токен с huggingface.co/settings/tokens
  "min_speakers": null,          // подсказки (необязательно)
  "max_speakers": null
}
```

Диаризация запускается автоматически после транскрипции — только для одиночных
дорожек. Если extra/токен не настроены, транскрипция не падает: стенограмма
сохраняется без меток спикеров, а в лог пишется причина. На GPU работает быстрее.

## Выжимка разговора (LLM, локально)

После транскрипции можно получить деловую выжимку: краткое содержание, ключевые
решения, задачи (action items) и открытые вопросы. По умолчанию — **локально**
через [Ollama](https://ollama.com) (звонок не покидает машину). Длинные стенограммы
сворачиваются по частям (map-reduce), чтобы не упереться в окно контекста.

### Установка Ollama (один раз)

```bash
# Linux
curl -fsSL https://ollama.com/install.sh | sh
ollama serve &                 # запустить сервер (если не как сервис)
ollama pull qwen2.5:7b         # модель ~4.7 GB (влезает в 8 GB VRAM)
```

На GTX 1080 (8 GB) выжимка идёт на GPU и занимает секунды-минуты — VRAM свободна
после транскрипции. Можно и на CPU (медленнее).

### Использование

- **Веб-интерфейс:** кнопка «Обработать (всё)» = транскрипция + выжимка, либо
  «Сделать выжимку» на вкладке «Стенограмма» (для уже готовой стенограммы).
- **CLI:** `python main.py summarize call_2026-05-30_09-20-00`

### Настройка (`summary` в `config.json`)

```jsonc
"summary": {
  "enabled": true,
  "provider": "ollama",          // "ollama" (локально) | "openai" | "none"
  "model": "qwen2.5:7b",
  "host": "http://localhost:11434",
  "num_ctx": 8192,
  "api_key": null,               // нужен только для provider="openai"
  "language": "ru"
}
```

Облако (`provider: "openai"`) — опционально, для него укажи `api_key`. Если хочешь
держать всё локально — оставь `ollama`. `provider: "none"` отключает выжимку.

## Экспорт в DOCX

Стенограмму и выжимку можно выгрузить одним документом Word:

- **Веб-интерфейс:** кнопка «Экспорт в DOCX» на вкладке «Стенограмма» —
  файл скачивается браузером (удобно и при работе через `ui-lan` с ноутбука).
- **CLI:** `python main.py export call_2026-05-30_09-20-00` → `transcripts/<stem>.docx`

В документ попадают шапка (файл, дата, длительность, язык, модель), выжимка
(если уже сделана) и реплики с таймкодами и метками спикеров.

## Упаковка для передачи

```bash
make dist           # callscribe.zip — только исходники (через git archive)
```

В архив попадают исходники, тесты, `templates/`, README и пустые `recordings/`/
`transcripts/`. **Не** попадают `config.json` (на каждой машине свои устройства),
`models/` (~3 GB, скачается сама) и записи — они в `.gitignore`. У получателя:

```bash
unzip callscribe.zip && cd callscribe
uv sync
uv run python main.py setup   # настроить аудио-устройства
uv run python main.py ui
```

## Структура

```
main.py          точка входа, CLI-роутер
recorder.py      захват аудио (sounddevice)
transcriber.py   STT + мультипроцессинг (faster-whisper)
ui.py            FastAPI + WebSocket
setup.py         мастер настройки устройств
config.py        работа с config.json и путями
templates/
  index.html     весь веб-интерфейс одним файлом
```
